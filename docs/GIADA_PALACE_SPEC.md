## 1. Product overview

### 1.1 What Giada Palace is

Accommodation facility in "zona ospedale" of Lucca, Italy — **15 minutes on foot from the historic walls**. Three unit types:

- **5 apartments with kitchen** (A, B, C, D, F — E is deliberately skipped, not a typo)
- **4 double rooms** (rooms 1–4)
- **2 single rooms** (rooms 5–6)

**Total: 11 bookable units.**

Seasonal outdoor pool (April–October), private parking, coral-pink façade, recently renovated interiors (black wrought-iron beds, navy velvet, rattan detailing).

### 1.2 Who it serves

- **Leisure** — families/couples on Tuscany vacation, May–September. Pool, space, kitchen autonomy, walkable Lucca.
- **Business** — workers in Lucca/Capannori, year-round. Parking, Wi-Fi, quiet neighborhood.

### 1.3 Primary goals

Brand presence, lead generation (form + WhatsApp + phone), SEO, language reach (IT+EN launch, DE+FR within 3 months).

### 1.4 Out of scope for v1

Online booking, payments, user accounts, blog, ZAK API integration.

## 5. Content model

### 5.1 Collections

Three collections in `src/content/`: `units/`, `amenities/`, `attractions/`.

### 5.2 `units/` Zod schema

```typescript
// src/content/config.ts
import { defineCollection, z } from 'astro:content';

const unitsCollection = defineCollection({
  type: 'content',
  schema: ({ image }) => z.object({
    slug: z.string(),
    name: z.object({
      it: z.string(),
      en: z.string(),
      de: z.string().optional(),
      fr: z.string().optional(),
    }),
    category: z.enum(['apartment', 'double', 'single']),
    order: z.number().int(),

    maxGuests: z.number().int().positive(),
    bedConfiguration: z.string(),
    size: z.number().positive().optional(),

    heroImage: image(),
    gallery: z.array(image()).min(3).max(12),

    features: z.object({
      it: z.array(z.string()),
      en: z.array(z.string()),
      de: z.array(z.string()).optional(),
      fr: z.array(z.string()).optional(),
    }),

    shortDescription: z.object({
      it: z.string().max(160),
      en: z.string().max(160),
      de: z.string().max(160).optional(),
      fr: z.string().max(160).optional(),
    }),

    seo: z.object({
      it: z.object({ title: z.string(), description: z.string() }),
      en: z.object({ title: z.string(), description: z.string() }),
      de: z.object({ title: z.string(), description: z.string() }).optional(),
      fr: z.object({ title: z.string(), description: z.string() }).optional(),
    }),

    availableOnRequest: z.boolean().default(true),
    seasonalNote: z.string().optional(),
    isDraft: z.boolean().default(false),

    // v1.2 addition: photo sharing metadata
    sharesPhotosWith: z.string().optional(),  // e.g. "apartment-f" → apartment-d shares with apartment-f
  }),
});

export const collections = { units: unitsCollection };
```

### 5.4 Photo-sharing pattern (apartments D and F)

Apartment D and F are twin units with identical layouts, so they share the same photo set. In the Content Collection files:

- `src/content/units/apartment-d.md` — frontmatter field `sharesPhotosWith: "apartment-f"`, gallery references point to `src/assets/images/units/apartment-df-shared/` folder
- `src/content/units/apartment-f.md` — frontmatter field `sharesPhotosWith: "apartment-d"`, gallery references point to the same folder

**Rules for Claude Code:**
1. **Never duplicate photo files.** The folder `apartment-df-shared/` is the single source of truth for both units.
2. The UI on apartment-d and apartment-f detail pages can optionally show a small note: "Appartamento gemello di F" / "Twin apartment to F" — decide at PR-5 review with Gennaro.
3. If in the future D or F is renovated and becomes visually different, split into separate folders and remove the `sharesPhotosWith` field.

### 5.5 File naming

- Content: `src/content/units/apartment-a.md`, `src/content/units/room-4.md`
- Shared photos folder: `src/assets/images/units/apartment-df-shared/hero.jpg`

### 5.6 Draft unit handling

All 8 non-representative units launch with `isDraft: true`:

- Page renders with minimal placeholder copy ("Informazioni dettagliate in arrivo. Contattaci per disponibilità.")
- `<meta name="robots" content="noindex, nofollow">` prevents SEO penalty
- Still linked from listing pages but with a subdued card style
- Contact form CTA works normally with `?unit=<slug>` pre-filled

### 11.2 Processing rules

Before committing processed images:
- Resize max 2400px long side
- JPEG quality 85 (visually lossless)
- Strip EXIF metadata
- Name: lowercase, hyphens, descriptive

Astro's `astro:assets` generates AVIF + WebP + JPEG at multiple sizes automatically.

### 11.3 Fallback images for units without photos

Units without dedicated photos (`apartment-c`, `room-1`, `room-2`, `room-3`, `room-6`) use a category-generic fallback in listing cards:

- Apartments without photos → `src/assets/images/units/generic-apartment.jpg` (a neutral shot from `apartment-a` gallery, slightly softened to not feel specific)
- Doubles without photos → `src/assets/images/units/generic-double.jpg` (from `room-4`)
- Singles without photos → `src/assets/images/units/generic-single.jpg` (from `room-5`)

On the detail page of these units, **gallery is hidden** and replaced by:
- A single fallback image at top
- Clear notice: "Foto dettagliate di questa unità in arrivo. Per vedere lo stile delle nostre camere, visita [Camera 4](/camere/room-4) o contattaci."
- Same features/description structure as representative units

This approach is honest with users (no fake photos) and maintains SEO (the page exists, noindex only for drafts).

### 11.4 Accessibility

Every image needs meaningful `alt` in display locale. Example: `alt="Piscina rotonda del Giada Palace al tramonto, con vista sul palazzo in rosa corallo"`. Never use `alt=""` unless truly decorative.
